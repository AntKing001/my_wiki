    grep命令

    grep命令用于查找文件里符合条件的字符串或者正则表达式。

    grep [options] pattern [files]

    wc命令

    wc用于计算字数，可以计算文件的字节数，字数，或者列数 知道名称，或者文件名为“—”，则从标准输入设备读取数据

    mv

    移动文件到新的位置，或者重命名文件。

    man

    查看命令使用手册，按q推出。

    man [命令]

    管道

    管道类似于文件重定向，可以将前一个命令的stdout重定向到下一个命令的stdin。

    特点： * 管道命令仅处理stdout，会忽略stderr。

    • 管道右边的命令必须能接受stdin。

    • 多个管道命令可以串联。

    与文件重定向的区别

    • 文件重定向左边为命令，右边为文件。

    • 管道左右两边均为命令，左边有stdout，右边有stdin。

## 进程通信方式
1. 管道(Pipe)：管道是一种半双工的通信方式，可以用于具有亲缘关系的父子进程之间的通信。

2. 命名管道(Named Pipe)：也称为FIFO，它允许无亲缘关系的进程通过文件系统中的一个特殊文件进行通信。

3. 信号(Signal): 通过向目标进程发送特定的信号，可以实现简单的异步通知和事件处理。

4.共享内存(Shared Memory): 共享内存是一种高效的进程间通信的方式，在多个进程之间共享同一块物理内存区域，从而避免了数据复制的开销。
在使用共享内存时，首先需要建立一个共享内存区域，并将其映射到各个进程的地址空间中。这样，不同的进程就可以通过访问相同的内存地址来读写数据，实现数据共享。当一个进程修改了共享内存中的数据时，其他进程也能立即看到变化。

5. 消息队列(Message Queue): 消息队列是一种基于消息传递的进程间通信方式，允许不同进程间通过发送和接受消息来实现数据交换。

6. 信号量(Semaphore):信号量对于实现共享资源的并发控制，通过增加或减少计数器来表示资源是否可用。

7. 套接字(Socket):套接字提供了一种在网络上进程间通信的机制，能够在不同主机的进程进行数据交换。


## Linux如何实现共享内存
1. 管道只能在具有亲缘关系的进程间进行通信，通过问价共享，在处理效率上又差一些，而且访问文件描述符不如访问内存地址方便。

2. mmap内存共享映射，mmap本来是存储映射功能，它可以将一个文件映射到内存当中，在程序里就可以直接使用内存地址对文件内容进行访问，Linux的mmap实现了一种可以在父子进程之间共享内存地址的方式。


## I/O复用
1. 阻塞I/O模型：在阻塞IO模型中，当应用发起一个IO操作后，程序会被阻塞，直到操作完成。这意味着应用程序在等待IO操作完成时无法执行其他的任务，造成CPU资源的浪费。

2. 非阻塞IO模型：非阻塞IO模型允许一个应用程序发起非阻塞IO操作后继续执行其他任务，而不必等待IO操作完成。应用程序可以通过轮训的方式检查IO操作是否完成，从而实现异步处理数据。

3. IO多路复用模型：IO多路复用模型时一种通过单线程同时监听多个IO时间的机制。常用的IO多路复用系统调用包括select，poll，epoll。通过IO多路复用，应用程序可以同时监听多个文件描述符（sockets），一旦其中任意一个文件描述符准备好IO操作，程序收到通知并且进行相应的处理，从而提高了并发性能。

### select

1. 单个进程可以监视的文件描述符存在最大限制，通常1024，采用轮训方式扫描文件描述符，文件描述符数量越多，性能越差。

2. 内核/用户空间内存拷贝问题，select需要大量句柄数据结构，产生巨大开销。

3. select返回的是整个句柄的数组，应用程序需要遍历整个数组才能发现哪些句柄发生事件。

4. select是采用水平触发，应用程序如果对没有完成对一个已经就绪的文件描述符进行IO操作，那么每次select调用还会将这些文件描述符通知进程。

### Poll

与select相比，Poll使用链表保持文件描述符，没有了监视文件数量的限制，但是其他缺点仍然存在。

### Epoll

`epoll_create`创建一个 epoll 对象
`epoll_ctl`向 epoll 对象中添加要管理的连接
`epoll_wait`等待其管理的连接上的 IO 事件

上面缺点都不存在，epoll使用一个文件描述符管理多个描述符。




## 线程池，内存池
1. 为什么需要线程池？
    大多数的网络服务器，包括 Web 服务器都具有一个特点，就是单位时间内必须处理数 目巨大的连接请求，但是处理时间却是比较短的。在传统的多线程服务器模型中是这 样实现的:一旦有个请求到达，就创建一个新的线程，由该线程执行任务，任务执行 完毕之后，线程就退出。这就是”即时创建，即时销毁”的策略。尽管与创建进程相 比，创建线程的时间已经大大的缩短，但是如果提交给线程的任务是执行时间较短， 而且执行次数非常频繁，那么服务器就将处于一个不停的创建线程和销毁线程的状态。 这笔开销是不可忽略的，尤其是线程执行的时间非常非常短的情况。

2. 线程池原理 
    在应用程序启动之后，就马上创建一定数量的线程，放入空闲的队列中。这些线程都 是处于阻塞状态，这些线程只占一点内存，不占用 CPU。当任务到来后，线程池将选 择一个空闲的线程，将任务传入此线程中运行。当所有的线程都处在处理任务的时候， 线程池将自动创建一定的数量的新线程，用于处理更多的任务。执行任务完成之后线 程并不退出，而是继续在线程池中等待下一次任务。当大部分线程处于阻塞状态时， 线程池将自动销毁一部分的线程，回收系统资源。

3. 线程池的作用 
    需要大量的线程来完成任务，且完成任务的时间比较短;对性能要求苛刻的应用;对性能要求苛刻的应用

4. 内存池的原理
    在软件开发中，有些对象使用非常频繁，那么我们可以预先在堆中实例化一些对象， 我们把维护这些对象的结构叫“内存池”。在需要用的时候，直接从内存池中拿，而 不用从新实例化，在要销毁的时候，不是直接 free/delete，而是返还给内存池。把那 些常用的对象存在内存池中，就不用频繁的分配/回收内存，可以相对减少内存碎片， 更重要的是实例化这样的对象更快，回收也更快。当内存池中的对象不够用的时候就 扩容。

5. 内存池的优缺点 
    内存池对象不是线程安全的，在多线程编程中，创建一个对象时必须加锁。
