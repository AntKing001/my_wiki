    C++

    编译过程

    1. 预处理-Pre-Prcocessing

    # -E 选项指示编译器仅对输入文件进行预处理
g++ -E test.cpp -o test.i

    2. 编译-Compiling

    # -s 编译选项告诉g++在为c++代码产生了汇编语言文件后停止编译。
# g++产生汇编语言文件到缺省扩展名是 .s
g++ -E test.i -o test.s

    3. 汇编-Assembling

    # -c 选项告诉g++仅把源代码编译为机器语言的目标代码。
# 缺省时，g++建立的目标代码文件有一个.o的扩展名。
g++ -E test.s -o test.o

    4. 链接-Linking

    # -o 编译选项来为将产生的可执行文件用指定的文件名
g++ test.o -o test

    CMake

    • 基本语法格式:指令1(参数1 参数2 参数3)

    • 指令大小写无关，参数和变量大小写相关

    set(HELLO hello.cpp)

add_executable(hello main.cpp hello.cpp)

ADD_EXECUTABLE(hello main.cpp ${HELLO})

    • 变量使用${}方式取值，但在IF控制语句中是直接使用变量名。

    • cmake——minimum_required 指定CMake的最小版本要求 

    # CMake最小版本要要求2.8.3

cmake——minimum——required(VERSION 2.8.3)
    • project - 定义工程名称，并可指定工程

    语法:project(projectname [cxx] [c] [java])

    # 指定工程名为HELLPWORD

project(HELLPWORD)
    • set 显示的定义变量 语法:set(VAR [value] [CACHE TYPE DOCSTRING [FORCE]] )
    # 定义src变量，其值为main.cpp hello.cpp

set(SRC sayhello.cpp hello.cpp)
    • include_directories 向工程添加多个特定的头文件搜索路径 -->相当于指定g++编译器参数的-l参数

    语法:include_directories([AFTER|BEFORE][SYSTEM] dir1 dir2...)

    # 将usr/include/myincludefolder 和 ./include添加到文件搜索路径

include_directories(/usr/include/myincludefolder ./include)
    • link_dorectories 向工程中添加多个特定的库文件搜索路径 -->相当于指定g++编译器参数的-L参数 语法:link_directories(dir1 dir2 ...)
    # 将/usr/lib/myfolder 和 ./lib添加到库文件搜索路径

link_directories(/usr/lib/myfolder 和 ./lib)

    面向对象三大特征

    封装：将具体实现过程和数据封装成一个函数，只能通过接口进行访问，降低耦合性，使类成为一个具有内部数据的自我隐藏能力、功能独立的软件模块。意义：保护或防止代码在无意之中被破坏，保护类中的成员，不让类中以外的程序直接访问或者修改，只能通过提供的公共接口访问。

    继承：子类继承父类的特征和行为，复用了基类的全体数据和成员函数，具有从基类复制而来的数据成员和成员函数（基类私有成员可被继承，但是无法被访问），其中构造函数、析构函数、友元函数、静态数据成员、静态成员函数都不能被继承。基类中成员的访问方式只能决定派生类能否访问它们。增强了代码耦合性，当父类中的成员变量或者类本身被final关键字修饰时，修饰的类不能被继承，修饰的成员变量不能重写或修改。意义：基类的程序代码可以被派生类服用，提高了软件复用的效率，缩短了软件开发的周期

    多态：不同继承类的对象对同一消息做出不同的响应，基类的指针指向或绑定到派生类的对象，使得基类指针呈现不同的表现形式。意义：对已存在的代码具有可替代性，对代码具有可扩充性，新增子类不会影响已存在类的各种性质，在程序中体现了灵活多样的操作，提高了使用效率，简化了对应用代码的编写和修改过程。

    final关键字的作用

    放在类的后面表示该类无法被继承，也就是阻止了从类的继承，放在虚函数后面该虚函数无法被重写，表示阻止虚函数的重载

    define和const的区别是什么

    编译阶段：define是在编译预处理阶段进行简单的文本替换，const是在编译阶段确定其值

    安全性：define定义的宏常量没有数据类型，只是进行简单的替换，不会进行类型安全检查；const定义的常量是有类型的，是要进行类型判断的

    内存占用：define定义的宏常量，在程序中使用多少次就会进行多少次替换，内存中有多个备份，占用的是代码段的内存；const定义常量占用静态存储区域的空间，程序运行过程中只有一份

    调试：define定义的宏常量不能调试，因为在预编译阶段就已经进行替换了；const定义的常量是可以进行调试的。

    new与malloc：

    new是一个运算符，而malloc是一个库函数。 new会自动计算所需要的内存空间，而malloc不可以 new会调用构造函数，而malloc不会。 new类型安全，而malloc不是 new/delete不需要库文件的支持，而malloc/free需要 new返回类型指针，而malloc返回*void类型指针，需要强制转换。 new运算符可以被重载，而malloc不可以

    多线程

    #include <iostream>
#include <thread>
using namespace std;
void fun(int x) {
    std::cout << "hello world" << x << endl;
}
int main () {
    thread t1(fun, 1);
    thread t2(fun, 2);
    t1.join();
    t2.join();
    return 0;
}

    智能指针

    #include<memory>

    unique_ptr: 作用域指针，超出作用域时自动销毁，然后调用delete。该指针不能进行复制，如果进行复制，两个指针unique_ptr指向同一快内存，若果其中一个死了，他会释放那段内存，另一个指针则会指向已经被释放的内存。 不能复制：拷贝构造函数与拷贝构造操作符被删除 

    unique_ptr(const _Myt&) = delete;
_Myt& operator=(const &_Myt&) = delete;

    class Entity {
public:
    Entity() {
        std::cout << "Create Entity" << std::endl;
    }
    ~Entity() {
        std::cout << "Destroy Entity" << std::endl;
    }
    void print() {}
};
int main () {
    {
        std::unique_ptr<Entity> entity(new Entity());
        std::unique_ptr<Entity> entity1 = std::make_unique<Entity>();  //异常安全(不会得到一个没有引用的悬空指针，从而造成内存泄露)

        std::unique_ptr<Entity> entity2 = entity;  //编译错误

        entity->print();
    }
    return 0;
}

    shared_ptr:通过引用计数，可以跟踪指针有多少的引用，一旦引用计数为0，就会删除。shared_ptr需要分配一块内存，叫做控制块，用来存储引用计数。多线程对于同一个shared_ptr实例的读操作可以保证线程安全，但对于同一个sharedptr实例的写操作（改变一个sharedptr指向的对象）则需要同步，否则会发生race condition。即sharedptr的引用计数本身是线程安全且无锁的，但sharedptr对象本身的读写则不是，因为sharedptr有两个数据成员，读写操作不能原子化。

    int main () {
    {
        std::shared_ptr<Entity> e0;
        {
            std::shared_ptr<Entity> entity = std::make_shared<Entity>();
            e0 = entity;
        }
        e0->print();
    }
    return 0;
}

    weak_ptr:可以赋值给weak_ptr，但不会增加引用计数

    int main () {
    {
        std::weak_ptr<Entity> e0;
        {
            std::shared_ptr<Entity> entity = std::make_shared<Entity>();
            e0 = entity;
            e0->print();    //错误，entity已经销毁，无法调用。
        }
    }
    return 0;
}

    GDB

    指令 功能
    run(r) 开始运行程序
    break(b) 设置断点，可以置顶文件名或行号或者参数名作为参数
    continue(c) 继续执行程序，知道下一个断点或者程序结束
    next(n) 执行下一行代码，并逐行执行函数调用（不会进入函数内部）
    step(s) 执行下一行代码，并逐行语句执行函数调用（会进入函数内部）

    backtrace(bt) 显示当前调用栈信息
    list(l) 显示当前位置附近的源代码
    info breakpoints 显示已设置的所有断点信息
    delete breakpoint 删除指定编号的断点
    watch 监视变量，在其值发生变化时自动停止程序运行
    quit(ctrl + D) 提出GDB调试器

    内存池

    主要解决以下问题：

    1. 内存碎片：频繁的进行动态内存分配和释放可能导致堆栈中出现碎片化空间，使得可用的连续内存块减少。使用内存池可以预先申请一块较大的连续内存空间，并将其划分为多个固定大小的小块，从而减少碎片化的问题。

    2. 内存分配时间开销：每次进行动态内存分配都需要操作系统介入并搜索合适的空闲内存区域，这会带来较大的时间开销。使用内存池，在程序启动时就可以预先申请一定量的内存，之后只需从内存池中获取已经分配好的对象，避免重复申请和释放造成的性能损失。

    3. 频繁申请和释放同样大小的对象：如果程序中有频繁地申请和释放同样大小的对象，使用内存池可以避免多次进行相同大小对象的动态分配与释放操作，提高效率。

    移动语义

    1. c++中的移动语意是一种特性，通过它可以在对象之间转移资源所有权，而不是进行复制。这个特效可以显著提高程序的效率与性能。

    2. 在传统c++中，当一个对象复制给另一个对象时，会执行深拷贝操作，将原始对象的值复制到新的对象中，这可能涉及大量数据复制和内存分配，对于大型对象来说非常耗费时间和资源。

    3. 而移动语意则允许将资源（如对上分配的内存，文件句柄等）从一个对象转移到另一个对象，而不进行数据复制，这是通过右值引用来实现的

    4. 在移动语意中，原对象被“盗取”了它拥有的资源，并且将其设置为无效状态。接受资源的目标对象获得了该资源并变为有效状态。这样就避免了数据复制和内存分配。

    5. 要使用移动语义，需要定义移动构造函数和赋值运算符，并使用std::move()函数来标记需要移动的资源。

    brpc

    brpc是一种基于c++的高性能rpc框架，相较于普通的c++线程，brpc框架的模型线程具有以下的优势：

    1. 轻量级：brpc使用协程而不是传统的系统线程模型，协程可用更轻量地创建和销毁，并且在切换上下文时开销更小。

    2. 高并发:brpc使用协程来处理请求，因为协程之间的切换成不很低，所以可以实现高并发处理请求。这对网络服务来说非常重要。

    brpc底层基于TCP/IP协议。它使用TCP作为传输层协议，通过建立可靠的连接来进行数据传输。在应用层上，brpc使用RPC协议进行通信。

    调用可执行程序

    Linux提供了system()函数和exec()函数，在c++程序当中，可以执行其他程序（二进制文件，操作系统命令或shell脚本）。

    exec函数族

    exec函数提供了另一种在进程中调用程序（二进制文件或shell脚本）的方法。

    1. 如果执行程序失败，直接返回-1，失败原因存与error中。

    2. 新进程的进程编号与原进程相同，但是新进程取代了原进程的代码段，数据段和堆栈。

    3. 如果执行成功函数不会返回，当在主程序中调用exec后，被调用的程序将取代调用者程序，也就是说，exec函数之后的代码都不会执行。

    I/O复用

    1. 阻塞I/O模型：在阻塞IO模型中，当应用发起一个IO操作后，程序会被阻塞，直到操作完成。这意味着应用程序在等待IO操作完成时无法执行其他的任务，造成CPU资源的浪费。

    2. 非阻塞IO模型：非阻塞IO模型允许一个应用程序发起非阻塞IO操作后继续执行其他任务，而不必等待IO操作完成。应用程序可以通过轮训的方式检查IO操作是否完成，从而实现异步处理数据。

    3. IO多路复用模型：IO多路复用模型时一种通过单线程同时监听多个IO时间的机制。常用的IO多路复用系统调用包括select，poll，epoll。通过IO多路复用，应用程序可以同时监听多个文件描述符（sockets），一旦其中任意一个文件描述符准备好IO操作，程序收到通知并且进行相应的处理，从而提高了并发性能。

    阻塞&非阻塞IO

    阻塞：在进/线程中，发起一个调用时，在调用返回之前，进/线程会阻塞等待，等待中的进/线程会让出CPU使用权。

    非阻塞：在进/线程中，发起一个调用时，会立即返回。

    回阻塞的四个函数：connect(), accept(), send(), recv()。

    在传统的网络服务端中，采用阻塞IO。

    在IO复用的模型中，事件循环不能阻塞在任何环节，所以，应该采用非阻塞IO。

    水平触发

    读事件：如果epoll_wait触发了读事件，表示有数据可读，如果程序没有把数据读完，再次调用epoll_wait的时候，将立即再次触发读事件。

    写事件：如果发送缓冲区没有满，表示可以写入数据，只要缓冲区没有写满，再次调用epoll_wait的时候，将再次触发写事件。

    边缘触发

    读事件：epoll_wait触发事件后，不管程序有没有处理读事件，epoll_wait都不会再触发读事件只有新数据到达时，才再次触发读事件。

    写事件：epoll_wait触发写事件后，如果发送缓冲区仍可以写（发送缓冲区没有满），epoll_wait不会再次触发写事件，只有当发送缓冲区由满变为不满时，才再次触发写事件。

    select模型

    1. 采用轮训的方式扫描bitmap，性能会随着socket数量增多而下降。

    2. 每次调用select()，需要拷贝bitmap。

    3. bitmap的大小（单个进/线程打开socket数量）由FD——SETSIZE宏设置，默认为1024个，可以修改，但是效率更低。

    poll模型（在select基础上进行了一些优化）

    1. 在程序中，poll的数据结构是数组，传入内核后转换成了链表。

    2. 在每调用一次select（）需要拷贝两次bitmap，poll拷贝一次结构体数组。

    3. poll监视的连接数没有1024的限制，但是，也是遍历的方法，监视的socket数量越多，效率越低。
